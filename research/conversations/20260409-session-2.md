## 1. BaseAgent API Surface — Resolved

### LLM Interaction
- `call_model(messages, **kwargs)` — raw chat completion
- `call_model_json(messages, schema, **kwargs)` — structured output, returns parsed object
- `call_model_stream(messages, **kwargs)` — streaming variant, yields chunks
- `call_model_validated(messages, validator_tool, **kwargs)` — first-class pattern: calls model, validates output via a tool, retries with backoff if invalid. Developers can extend with domain-specific validators (e.g., SQL syntax, schema compliance).

### Tool Use (Two Planes)
- `use_tool(name, **kwargs)` — call any tool from agent code (plane 1). Logging, RBAC, retry happen automatically.
- `register_tool(tool, visibility)` — register with visibility: `agent_only`, `llm_only`, or `both`
- `list_tools()` — returns available tools with metadata
- `get_llm_tools()` — returns tool definitions formatted for LLM tool calling API (only `llm_only` and `both`)
- `handle_tool_call(tool_call)` — routes LLM-initiated tool calls through the same logging/RBAC/retry infrastructure

### Two Tool Planes
- **Plane 1 (agent-code tools):** Called by agent Python code. Things like `open_door`, `email_user`, `validate_schema`. LLM never sees them. Used for structured actions with logging and RBAC.
- **Plane 2 (LLM-callable tools):** Surfaced to the LLM for tool calling. MCP tools land here by default. Some local tools can be surfaced here too.
- **Both planes go through BaseAgent** — logging, RBAC, retry, rate limiting all happen regardless of who initiated the call.
- **Visibility attribute per tool:** `agent_only`, `llm_only`, `both`

### Prompts
- `load_prompt(name, **variables)` — loads markdown file, substitutes variables
- `list_prompts()` — returns available prompts with metadata

### MCP Client (FastMCP v3)
- `connect_mcp(url)` — connects, discovers and registers tools
- `disconnect_mcp(url)` — clean disconnection
- Discovered MCP tools default to `llm_only` visibility

### Memory (via MemoryHub SDK)
- `self.memory` — MemoryHub SDK client, available when `.memoryhub.yaml` is present
- `self.memory.write(content, scope, weight)` — programmatic memory storage
- `self.memory.search(query, max_results)` — programmatic memory retrieval
- `None` when MemoryHub is not configured

### Skills (agentskills.io spec)
- `self.skills` — dict of loaded stubs (name → description)
- `load_skill(name)` — loads full SKILL.md body into context
- `unload_skill(name)` — drops back to stub, reclaims context

### Lifecycle
- `setup()` — loads config, connects MCP servers, discovers tools/prompts, initializes memory if configured, loads skill stubs
- `step()` — abstract, subclass implements. The main loop body.
- `teardown()` — disconnects MCP, closes memory session, cleanup
- `run()` — the loop driver. Calls setup(), loops step() until done, calls teardown(). Built-in max iterations, exponential backoff/retry.

### Conversation/State
- `messages` — conversation history
- `add_message(role, content)` — append
- `clear_messages()` — reset

### Config
- `config` — loaded agent.yaml

### Protective Patterns (everywhere relevant)
- Max iterations on the loop
- Exponential backoff/retry on model calls and tool calls
- Rate limiting
- Context budget awareness (future consideration)

## 2. Tool Architecture — Resolved

### Local Tools Live With the Agent
- Tools in `tools/` directory, one file per tool, auto-discovered
- For now, tools are per-agent. No central registry yet.
- Each tool has a visibility attribute controlling whether it's agent-only, LLM-only, or both.

### Tool Hub — Future Side Quest
- Enterprise tool registry living in OpenShift AI
- Per-tool RBAC with agent permissions
- Full logging and audit trail
- Tool quarantine capability (revoke access to a defective tool across all agents)
- Agent quarantine (disable a bad agent)
- Forensic trace: agent_id → driver_id → tool_id
- Captured as a side quest artifact, not in scope for this project.

### Anti-Corruption / Validation Pattern
- `call_model_validated()` is a first-class BaseAgent method
- Get LLM output → validate with a tool → retry with feedback if invalid
- Extensible: developers can build domain-specific validators
- Enables patterns like code execution sandboxes where LLM generates code and a sandboxed tool executes it

## 3. Agent Config Format — Resolved

```yaml
# agent.yaml
model:
  endpoint: ${MODEL_ENDPOINT:-http://llamastack:8321/v1}
  name: ${MODEL_NAME:-meta-llama/Llama-3.3-70B-Instruct}
  temperature: 0.7
  max_tokens: 4096

mcp_servers:
  - url: ${MCP_WEATHER_URL:-http://weather-mcp:8080/mcp}

tools:
  local_dir: ./tools
  visibility_default: agent_only

prompts:
  dir: ./prompts

loop:
  max_iterations: ${MAX_ITERATIONS:-100}
  backoff:
    initial: 1.0
    max: 30.0
    multiplier: 2.0

logging:
  level: ${LOG_LEVEL:-INFO}
```

- YAML with `${VAR:-default}` env var substitution
- Same config works locally and on OpenShift (different env vars per environment)
- Helm chart's ConfigMap mounts this file; values.yaml sets env vars
- When `.memoryhub.yaml` is present, MemoryHub MCP server is auto-added (no manual entry needed)

## 4. AGENTS.md — Resolved

- Follows the open standard at agents.md
- Just markdown, no required fields, no strict schema
- For our purpose-built agents, mostly minimal or unnecessary
- The agent's behavior comes from: system prompt (prompts/system.md), agent.yaml, skills, rules
- AGENTS.md is for project-level overrides — stuff you'd tell a new teammate
- Scaffolded as a minimal file with guidance comments

## 5. Skills — Resolved

- Follow the agentskills.io specification exactly
- Each skill is a directory containing SKILL.md (required) plus optional scripts/, references/, assets/
- Progressive disclosure: frontmatter (~100 tokens) loaded at startup, full body on activation, resources on demand
- SKILL.md has YAML frontmatter (name, description required; license, compatibility, metadata, allowed-tools optional) + markdown body
- Name must match parent directory name, lowercase with hyphens only
- Keep SKILL.md under 500 lines, split larger content into references/
- No commands — skills replace commands entirely

## 6. Evals Scaffold — Resolved

- `evals/` directory with README, evals.yaml, and run_evals.py
- evals.yaml defines cases in harness-agnostic format (input, expected, tags)
- Lightweight local runner in run_evals.py
- Designed to be exportable to external harnesses (Inspect AI, OpenAI evals, etc.)

## 7. MemoryHub Integration — Resolved

### Dual-Path Memory Access
- **MCP path (LLM use):** MemoryHub's 15 tools available through MCP client. LLM calls memory_search, memory_write, etc. during reasoning. Rules file guides when.
- **SDK path (agent-code use):** `self.memory` exposes the MemoryHub Python SDK. Agent code can programmatically store/retrieve memories. SDK handles OAuth automatically.

### Developer Experience
1. Scaffold agent with `fips-agents create agent my-agent`
2. Run `memoryhub config init` in the agent directory
3. Wizard generates `.memoryhub.yaml` (behavioral config, committed) + rules file in `rules/`
4. MemoryHub MCP server auto-added to agent.yaml
5. Agent subclass doesn't change

### BaseAgent Behavior
- When `.memoryhub.yaml` present: initializes SDK client (`self.memory`), connects MCP, registers session, loads memories per loading pattern, handles topic shift detection
- When absent: `self.memory` is None, no MCP connection, agent works fine without it

### Multi-Agent / Hive Mind
- Multiple agents connect to same MemoryHub instance
- Scope controls visibility (user/project/role/org/enterprise)
- RBAC controls write permissions
- Agents share context without direct communication
- Use case examples: healthcare operations, emergency response, swarms

### Immutability
- `.memoryhub.yaml` is part of the container image (same as prompts, tools, code)
- OAuth credentials come from OpenShift Secrets
- Endpoint URL comes from agent.yaml ConfigMap

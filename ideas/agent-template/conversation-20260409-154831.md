## How fips-agents Invokes Templates — Resolved

The `fips-agents` CLI clones a public template repo. The template includes a `.claude/` directory with:
- Slash commands that guide the developer through a structured workflow
- Rules that enforce good practices
- CLAUDE.md that sets context for the AI assistant

This is the same pattern used by the MCP server template at `redhat-ai-americas/mcp-server-template`. The developer experience IS the template.

Reference: The MCP server template uses `/plan-tools` → `/create-tools` → `/exercise-tools` → `/deploy-mcp` as its workflow.

## Slash Commands — Resolved

### Core Workflow (linear sequence)
- `/plan-agent` — Design the agent: define purpose, tools, prompts, skills. Produces AGENT_PLAN.md. No code.
- `/create-agent` — Reads AGENT_PLAN.md, scaffolds agent subclass, tools, prompts, skills, rules. Implements in parallel with subagents.
- `/exercise-agent` — Role-plays as a user/caller to test agent behavior, tool usage, edge cases. Structured feedback.
- `/deploy-agent` — Pre-flight checks, builds container, deploys to OpenShift via Helm, verifies.

### Utility Commands
- `/add-tool` — Scaffold a new tool with the @tool decorator
- `/add-skill` — Scaffold a new skill directory following agentskills.io spec
- `/add-memory` — Runs memoryhub config init and wires in the MemoryHub integration

## Tool Auto-Discovery with Decorators — Resolved

Decorator convention, same as FastMCP:

```python
from base_agent.tools import tool

@tool(visibility="agent_only")
async def validate_schema(data: dict, schema_name: str) -> bool:
    """Validate data against a named schema."""
    ...
```

- BaseAgent auto-discovers all @tool decorated functions in tools/
- Name from function name, description from docstring, params from type hints
- Visibility from decorator argument: agent_only (default), llm_only, or both
- No registration boilerplate

## Rules Format — Resolved

Plain markdown files in `rules/` directory. No frontmatter — filename is the identifier, content is the guidance.

Example (from memory-hub's memoryhub-integration.md):
- Clear headings for sections (Session Setup, During Work, Memory Hygiene)
- Imperative instructions
- Practical, actionable guidance
- BaseAgent loads all .md files from rules/ and injects into context

## BaseAgent Implementation — Resolved

### No Framework
- Pure Python, async throughout
- No LangChain, no LangGraph, no framework dependencies
- Direct control over the agent loop

### LLM Client: litellm
- litellm as the LLM client layer (not raw openai SDK, not httpx)
- 100+ providers supported: vLLM, LlamaStack, Anthropic, OpenAI, Azure, Bedrock, etc.
- OpenAI-compatible interface with tool calling, structured output, streaming
- Swap providers by changing the model string prefix in config
- 42.7k GitHub stars, actively maintained
- One dependency that gives us full provider portability

### Why litellm over openai SDK
- openai SDK only speaks OpenAI wire format — works with vLLM and LlamaStack but not Anthropic, Bedrock, etc.
- litellm wraps all providers behind a single interface
- Model string like "anthropic/claude-sonnet-4-20250514" or "openai/gpt-4" or just "meta-llama/Llama-3.3-70B-Instruct" for vLLM
- Same completion() call regardless of provider

### Other Dependencies
- FastMCP v3 — MCP client
- memoryhub SDK — optional, for MemoryHub integration
- pydantic — config validation, structured output schemas
- httpx — async HTTP (used by litellm and FastMCP internally)
- python-frontmatter — parsing prompt and skill files
- Standard library for everything else

## Workflow Manager — Resolved (Design Direction)

LangGraph-like state graph without the package dependency. Custom implementation.

Key concepts:
- Nodes are BaseAgent subclasses (each with their own step())
- Edges are conditional routing based on node output
- State is a typed dict that flows through the graph
- Manager handles state passing between nodes
- Supports conditional branching and cycles
- Checkpointing for resume-after-failure

Deferred for build, but BaseAgent's design supports it — each node is just a BaseAgent subclass.
